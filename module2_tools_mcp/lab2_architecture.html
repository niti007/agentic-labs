📦What is MCP?

The Model Context Protocol — a standard for wrapping tools so Claude can call them:

  • Small Python programs
  • Expose named tools with descriptions
  • Return structured results
  • Claude Code auto-discovers them

The delivery standard — not a toy simulator.

💡 WHY This Matters

🎯Interfaces are Everything

The model only sees tool names and descriptions when deciding which tool to call. Clear names = correct selection. Vague names = wrong tool, failed task.

⚠️Errors Drive Recovery

When a tool fails, the agent needs to know: is this transient (retry) or permanent (stop)? Without structured errors, the agent flails or gives up on fixable problems.

🛡️Scope Prevents Mistakes

Give an agent exactly one tool and it can't misbehave. Scoping is how you make a classification step deterministic — not by hope, but by physics.

🎓 Core Lesson: An AI agent is only as reliable as its tools. The three design choices — clear interfaces, structured errors, and tight scoping — are not optional. They live in the code you author (MCP servers, CLAUDE.md policies, subagent tool lists), and Claude Code runs the selection and retry logic. This is how you go from "kinda works" to "ships to production."

🔗 HOW Everything Connects

How the Connection Works:

  1. User asks a question → Claude Code receives it
  2. Claude looks at all available tools → reads from .mcp.json file
  3. .mcp.json points to MCP servers → small Python programs you write
  4. Each server defines tools → with @mcp.tool() decorators (names + docstrings)
  5. Claude reads tool names/descriptions → that's ALL it sees to decide which tool
  6. Claude calls the matching tool → tool runs in the MCP server process
  7. Tool returns result → with structured fields (isError, isRetryable, content, etc.)
  8. CLAUDE.md policy dictates next step → retry if retryable, halt if not
  9. For scoped steps → subagent's tools: list restricts what's available

📺 Lab 2.1 — Three Design Patterns (Demos G–I)

✓ Demo G: Clear Interfaces

What: Ask "Where is my order?" → Claude calls search_orders. Ask "Do you sell keyboards?" → Claude calls search_products.

Why: Same domain (shopping), but tool names are so clear the model picks correctly every time.

How: The only info Claude has is the tool name and docstring. Good naming → correct selection.

Failure mode: Swap to search_a / search_b and Claude guesses wrong or asks which one you mean.

✓ Demo H: Structured Errors

What: Call get_order with a valid ID → first call times out (504) → Claude retries → succeeds. Call with missing ID → returns 404 → Claude stops.

Why: Without structured errors, Claude either retries forever or gives up. With isRetryable flag, it recovers from transients but stops on permanents.

How: Tool returns { isError: true, isRetryable: true/false, ... }. CLAUDE.md policy says: if retryable, try again.

Principle: Transient 504 → success on retry. Permanent 404 → clean halt.

✓ Demo I: Scoped Access

What: Route three texts to a classifier subagent. Each produces only a label tool call (sentiment + confidence). No prose, no wandering.

Why: You want clean, deterministic output. By giving the step exactly one tool, you guarantee the shape.

How: Subagent's CLAUDE.md lists tools: mcp__shop__label. That's the only tool it can see or call.

Principle: Scope enforces shape. The agent can't go off-script if the script is the only thing in the room.

🔑 Key Point — Demo G: The model has nothing to go on but the tool name and description when deciding which tool to call. Two well-named tools → correct every time. Two vague ones → wrong guess. Write clear interfaces; they are the contract.
🔑 Key Point — Demo H: When a tool fails, return structured error fields: isError (did it fail?) and isRetryable (transient or permanent?). The policy in CLAUDE.md tells Claude what to do. Retry? Stop? This design is what makes the agent resilient.
🔑 Key Point — Demo I: If you want a step to produce one specific output shape, give it exactly one tool. Scoping is how you make deterministic behavior — not by hoping the agent is careful, but by physics: no other tool to call.

📺 Lab 2.2 — Multi-Source Tools (Demos J–L)

✓ Demo J: Two MCP Servers

What: .mcp.json wires both a database server (query_db) and a docs server (search_docs). Ask "Who is user 42?" → query_db. Ask "How do I configure X?" → search_docs.

Why: Real work pulls from multiple sources. Show the agent what's available; it picks the right source.

How: One .mcp.json file lists multiple servers; each has its own tools with distinct names.

✓ Demo K: Built-In Tools

What: Claude Code also offers built-in file tools: Glob (find files), Grep (search), Read (open file), Edit/Write. Use these to explore a codebase surgically.

Why: Before changing code, find the exact file and read only what you need — not dump everything.

How: Use Glob to find *.test.ts files, then Grep to search one keyword, then Read the specific file.

✓ Demo L: Incremental Exploration

What: Grep "validateToken" → Read auth.ts (not the whole repo) → Edit one line. Narrow, don't dump.

Why: Precise, fast, less risk. Dumping the whole repo and changing everywhere is slow and fragile.

How: Chain tool calls: find → read targeted → edit specific. Each tool narrows the scope.

🔑 Principle — Lab 2.2: Real work scales by connecting multiple data sources (DB, docs, code). Good tool design means the agent can route to the right source and narrow its search — not by hard-coded routing, but by clear interfaces and structured results.

📦 What is MCP? (Quick Reference)

1MCP Server: A small Python program that defines tools using @mcp.tool() decorators.

2Each tool is a function with a name, docstring, and return type — the name and docstring are what Claude sees.

3.mcp.json points to servers: Lists which Python files to run, and the server names they'll expose.

4Claude Code loads it: On startup, Claude Code reads .mcp.json, starts the MCP server processes, and asks you to approve them.

5Claude sees the tools: Once approved, the tools are available as if they were built-in. Claude calls them by name; the MCP server handles the execution.

6Results flow back: Tool returns data → Claude reads it → decides next step (retry, halt, or continue).

Example Tool Definition (in Python):

@mcp.tool() def search_orders(customer_id: str, date_range: str = "30d") -> dict: """Search for a customer's past purchases by ID or date range. Use this to answer 'Where is my order?' questions.""" # Implementation here return { "orders": [...], "status": "success" }

What Claude sees: Name = search_orders, Description = the docstring. That's all.

🔗 How Tool Design Connects to Real Work

🎯 Demo G — Clear Interfaces: In production, you have a customer support chatbot pulling from orders and product databases. If the tool names are vague (search_catalog / search_history), the agent mixes them up. If they're specific (search_orders / search_products), it routes correctly every time.
🎯 Demo H — Structured Errors: Your payment API sometimes times out. Without structured errors, the agent retries once and fails. With isRetryable:true, it retries automatically. A missing order (404) returns isRetryable:false, so it stops without wasting calls.
🎯 Demo I — Scoped Access: You're auto-scoring leads. You want sentiment always returned as {positive/neutral/negative, confidence 0-1}, never prose. Giving the scorer only the label tool guarantees that shape — no escape routes.
🎯 Lab 2.2 Multi-Source: A real system pulls from the billing database, the knowledge base, and the codebase. Each has different queries. With clear tool names (query_db, search_docs, Grep), the agent routes to the right source without ambiguity.

🎓 Key Takeaways

1️⃣Interfaces Drive Selection

The model only reads tool names and descriptions to decide which tool to call. Everything else is invisible. Write clear interfaces; they are literally the only thing Claude sees.

2️⃣Structured Errors Enable Recovery

Return error fields (isError, isRetryable) so Claude knows the difference between transient failures (retry) and permanent ones (halt). This turns flaky systems into resilient ones.

3️⃣Scope Guarantees Shape

Give a step exactly one tool and it can't misbehave. Scoping is how you make deterministic output — not by hope, but by architecture. The agent can't go off-script if the script is the only thing in the room.

4️⃣MCP is the Standard

MCP servers are not toy simulators — they're the real delivery mechanism. What you build here in Python with @mcp.tool() is what you'd ship to production.

5️⃣Multi-Source Scales Real Work

Connect databases, docs, code, and APIs in one .mcp.json file. The agent routes to the right source using clear tool names — no hard-coded routing needed.

6️⃣Narrow, Don't Dump

Use Glob → Grep → Read to find the exact file and read only what you need. Dumping everything is slow, risky, and wastes tokens. Surgical exploration scales.
