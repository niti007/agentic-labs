💡 WHY This Matters

⚡Reliability

AI assistants must keep going until a task is truly done, not stop halfway or guess. The agentic loop teaches this fundamental behavior.

🎯Specialization

Splitting big jobs into smaller, focused specialists makes the system more robust, testable, and easier to debug than one giant agent trying to do everything.

🔒Safety

Before you trust an AI with real work, you need guardrails (blocking bad actions), the right freedom level (fixed vs. flexible), and undo/branch capability.

🎓 Core Lesson: An AI assistant is like a new team member — you need to teach it to keep working until done, split big jobs into small pieces, stay within safe boundaries, and explore reversibly. These are the foundations that make AI trustworthy for real work.

🔗 HOW Everything Connects

How the Parts Connect:

  • User Question → triggers the lead agent in CLAUDE.md
  • Lead Agent → reads house rules + looks at available agents in .claude/agents/
  • Pattern Decision → uses commands in .claude/commands/ (like /pipeline) OR calls subagents directly
  • Specialist Agents → each file in .claude/agents/ is a focused assistant for one job
  • Tool Use → agents use Glob/Grep/Read/Edit/Write or tools from .mcp.json
  • Protection → .claude/hooks/ intercepts tool calls before they happen (PreToolUse)
  • Logging → .claude/hooks/ records what happened (PostToolUse)
  • Practice Data → notes/, invoices/, tickets/ give the agents real-looking work to do

📺 The Six Demos (A–F)

✓ Demo A: The Loop

What: Ask "Find the rate limit policy." The agent loops: read file → not enough info → read another → loop again → success → stop.

Why: Shows the core: "keep trying until you have a real answer."

How: Claude Code's tool_use (keep going) → end_turn (stop) switching.

✓ Demo B: Hub-and-Spoke

What: Ask "Summarize this, then translate it, then validate it." One lead agent routes each job to a specialist in .claude/agents/.

Why: Shows specialization: each agent does one thing well.

How: Lead agent calls the Task tool with agent name + prompt → specialist runs → result comes back.

✓ Demo C: Ordered Pipeline

What: Type /pipeline. Step 1 (extractor) runs → guard checks output → if empty, stop → else Step 2 (analyzer) runs.

Why: Shows reliability: bad input is caught before it cascades.

How: Command file defines the steps; each step waits for the previous result before starting.

✓ Demo D: Guardrails (Hook)

What: Ask "Write a secret to protected/secrets.env." The PreToolUse hook blocks it instantly, explains why.

Why: Shows safety: sensitive folders can't be touched, even by accident.

How: .claude/hooks/guard.py runs before any file write and checks the path.

✓ Demo E: Fixed vs. Flexible

What: Type /invoice-flow (always same 3 steps) vs. /triage (different steps per ticket type).

Why: Shows control: sometimes you want a recipe, sometimes you need decision-making.

How: Two command files — one lists fixed steps, one branches on input.

✓ Demo F: Fork & Resume (Terminal)

What: claude --resume <id> --fork-session in terminal — explore Solution A from a checkpoint, then go back and try Solution B.

Why: Shows reversibility: risky work can be explored safely without losing the base.

How: Claude Code's session save/fork feature; run from command line.

🔗 How the Patterns Connect to Real Work

🎯 The Loop: Every task that needs multiple steps to gather enough info to answer. Example: "What is our data retention policy?" → Loop: search config → read policy file → check related docs → answer.
🎯 Hub-and-Spoke: When a task can be split into independent subtasks. Example: Lead agent gets text → routes to Summarizer → routes to Translator → routes to Validator → assembles three outputs. Each specialist is replaceable.
🎯 Ordered Pipeline: When steps depend on each other AND you need to guard against bad input flowing forward. Example: Extract data → Validate structure → Analyze content. If Extract is empty, stop before trying to Analyze nonsense.
🎯 Guardrails: When you have sensitive files or operations. Hook runs *before* the action (PreToolUse = safe), or logs *after* the action (PostToolUse = audit). Never wait for the AI to "be careful" — the system prevents mistakes.
🎯 Fixed Recipe vs. Flexible Triage: Fixed recipe for routine, predictable work (same 3 steps every time). Flexible triage for judgment calls (different paths per case).
🎯 Fork & Resume: For risky or exploratory work — save a checkpoint, try one approach, go back and try another, compare. No work is lost.

🎓 Key Takeaways

1️⃣The Loop

An AI doesn't "know" — it tries, checks, and loops until done. This is the core habit. Everything else builds on it.

2️⃣Specialization

One big agent is fragile. Split into focused specialists that can be tested, replaced, and debugged separately. Hub-and-spoke is the pattern.

3️⃣Guardrails First

Never trust the AI to be careful. Build safety into the system itself: hooks block bad actions *before* they happen, not after.

4️⃣Files = Behavior

CLAUDE.md, agents, commands, hooks, and skills all together define how Claude behaves. Change a file, change the behavior — no code rewrite.

5️⃣Reversibility

Before you let an AI act on real work, build in save points, branches, and undo. Forks let you explore without losing your base.

6️⃣Plan First

For risky work, use Plan mode: AI writes a plan, you approve it, only then it edits. Prevents costly surprises.
